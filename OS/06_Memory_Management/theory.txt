================================================================================
                    MEMORY MANAGEMENT
================================================================================

Memory management handles allocation and deallocation of memory to processes.


================================================================================
                    1. MEMORY HIERARCHY
================================================================================

Speed/Cost                              Capacity
    ↑                                       ↑
    │   +------------+                      │
    │   │ Registers  │  ← Fastest, smallest │
    │   +------------+                      │
    │   │   Cache    │  ← L1, L2, L3        │
    │   +------------+                      │
    │   │Main Memory │  ← RAM               │
    │   │   (RAM)    │                      │
    │   +------------+                      │
    │   │   Disk     │  ← Secondary storage │
    │   +------------+                      │
    ↓   │   Tape     │  ← Backup            ↓


LOCALITY OF REFERENCE:
- Temporal: Recently accessed data likely accessed again
- Spatial: Nearby data likely accessed soon


================================================================================
                    2. BASIC CONCEPTS
================================================================================

LOGICAL ADDRESS (Virtual Address):
- Generated by CPU
- Used by process
- What process sees

PHYSICAL ADDRESS:
- Actual address in memory
- Used by memory unit
- Where data actually is


ADDRESS BINDING:
When logical addresses mapped to physical addresses.

1. Compile Time:
   - Physical address known at compile time
   - Must recompile if location changes
   - Used in embedded systems

2. Load Time:
   - Physical address determined when loaded
   - Relocatable code

3. Execution Time (Runtime):
   - Address binding delayed until execution
   - Most flexible
   - Needs hardware support (MMU)


MEMORY MANAGEMENT UNIT (MMU):
Hardware device that maps logical to physical addresses.

    CPU ──→ Logical ──→ [MMU] ──→ Physical ──→ Memory
            Address                Address


================================================================================
                    3. CONTIGUOUS MEMORY ALLOCATION
================================================================================

Each process gets single contiguous block of memory.


MEMORY PARTITIONING:

1. FIXED (STATIC) PARTITIONING:
   - Memory divided into fixed-size partitions
   - Each partition holds one process

   +------------------+
   |    OS Kernel     |
   +------------------+
   |   Partition 1    |  ← Process A
   +------------------+
   |   Partition 2    |  ← Process B
   +------------------+
   |   Partition 3    |  ← Empty
   +------------------+

   Problems:
   - Internal Fragmentation: Unused space within partition
   - Limited multiprogramming: Fixed number of partitions


2. VARIABLE (DYNAMIC) PARTITIONING:
   - Partition size based on process need
   - No internal fragmentation

   +------------------+
   |    OS Kernel     |
   +------------------+
   |   Process A      |
   +------------------+
   |      Hole        |  ← External Fragmentation
   +------------------+
   |   Process B      |
   +------------------+
   |      Hole        |
   +------------------+

   Problem:
   - External Fragmentation: Free memory in non-contiguous holes


================================================================================
                    4. ALLOCATION STRATEGIES
================================================================================

Given: Request for n bytes, multiple free holes.


1. FIRST FIT:
   - Allocate first hole big enough
   - Fast, simple
   - May leave small holes at beginning

   Example: Holes [100KB, 500KB, 200KB, 300KB], Request 210KB
   → Allocate from 500KB hole


2. BEST FIT:
   - Allocate smallest hole big enough
   - Minimizes leftover space
   - Leaves very small holes, slow (search all)

   Example: Holes [100KB, 500KB, 200KB, 300KB], Request 210KB
   → Allocate from 300KB hole (smallest that fits)


3. WORST FIT:
   - Allocate largest hole
   - Leaves larger leftover (potentially useful)
   - Slow (search all)

   Example: Holes [100KB, 500KB, 200KB, 300KB], Request 210KB
   → Allocate from 500KB hole


COMPARISON:
- First Fit: Generally fastest and good enough
- Best Fit: Most space efficient but creates tiny holes
- Worst Fit: Usually worst performance


================================================================================
                    5. FRAGMENTATION
================================================================================

INTERNAL FRAGMENTATION:
- Allocated memory larger than requested
- Unused space inside allocated block
- Occurs with fixed partitioning

   Process needs 18KB, partition is 20KB
   Internal fragmentation = 2KB


EXTERNAL FRAGMENTATION:
- Total free memory sufficient but not contiguous
- Holes scattered throughout memory
- Occurs with variable partitioning

   Free: 100KB + 200KB + 50KB = 350KB total
   Request: 300KB → Cannot allocate!


COMPACTION:
Solution for external fragmentation.
- Relocate processes to create contiguous free space
- Expensive (requires moving data)
- Only possible with runtime binding


Before:                    After:
+--------+                 +--------+
| OS     |                 | OS     |
+--------+                 +--------+
|  P1    |                 |  P1    |
+--------+                 +--------+
|  Hole  |                 |  P2    |
+--------+                 +--------+
|  P2    |                 |  P3    |
+--------+                 +--------+
|  Hole  |                 |  Hole  |  ← All free space
+--------+                 | (large)|
|  P3    |                 +--------+
+--------+


================================================================================
                    6. PAGING
================================================================================

Non-contiguous memory allocation.
Divides memory into fixed-size blocks.

TERMS:
- Frame: Fixed-size block of physical memory
- Page: Fixed-size block of logical memory
- Page size = Frame size (power of 2, e.g., 4KB)


HOW IT WORKS:

Logical Memory:          Physical Memory:
+--------+               +--------+
| Page 0 |────────────→  | Frame 2| (Page 0)
+--------+               +--------+
| Page 1 |────────+      | Frame 0|
+--------+        |      +--------+
| Page 2 |───+    +────→ | Frame 3| (Page 1)
+--------+   |           +--------+
| Page 3 |─+ |           | Frame 1|
+--------+ | |           +--------+
           | +─────────→ | Frame 4| (Page 2)
           |             +--------+
           +───────────→ | Frame 5| (Page 3)
                         +--------+


PAGE TABLE:
Maps pages to frames.
Each process has its own page table.

Page Number | Frame Number
    0       |      2
    1       |      3
    2       |      4
    3       |      5


ADDRESS TRANSLATION:

Logical Address = Page Number + Page Offset

If page size = 2^n bytes:
- Lower n bits = Page Offset (position within page)
- Upper bits = Page Number

Example:
Page size = 4KB = 4096 = 2^12
Logical address = 8196 (decimal)

Page Number = 8196 / 4096 = 2
Offset = 8196 % 4096 = 4

Page 2 → Frame 4 (from page table)
Physical Address = Frame 4 × 4096 + 4 = 16388


NO EXTERNAL FRAGMENTATION:
Any free frame can be allocated to any page.

INTERNAL FRAGMENTATION:
Average = half of page size (last page may not be full).


================================================================================
                    7. PAGE TABLE IMPLEMENTATION
================================================================================

1. PAGE TABLE BASE REGISTER (PTBR):
   - Points to page table in memory
   - Context switch: change PTBR

   Problem: Two memory accesses (page table + actual data)


2. TRANSLATION LOOK-ASIDE BUFFER (TLB):
   - Fast associative memory (cache)
   - Stores recent page-frame mappings
   - Hit: One memory access
   - Miss: Two memory accesses + update TLB

   TLB Structure:
   | Page # | Frame # | Valid | ASID |

   ASID: Address Space Identifier (process ID)
   Allows TLB to hold entries from multiple processes.


EFFECTIVE ACCESS TIME (EAT):

EAT = Hit_ratio × (TLB_time + Memory_time) +
      (1 - Hit_ratio) × (TLB_time + 2 × Memory_time)

Example:
TLB access = 10ns, Memory access = 100ns, Hit ratio = 90%

EAT = 0.9 × (10 + 100) + 0.1 × (10 + 200)
    = 0.9 × 110 + 0.1 × 210
    = 99 + 21 = 120ns


================================================================================
                    8. PAGE TABLE STRUCTURES
================================================================================

Problem: Page table itself can be very large.

32-bit address, 4KB pages, 4-byte entries:
- Pages = 2^32 / 2^12 = 2^20 = 1 million pages
- Page table size = 1M × 4B = 4MB per process!


SOLUTIONS:

1. HIERARCHICAL PAGING (Multi-level):

Two-Level Page Table:

    +------------+
    |Outer Page  |      +------------+
    |  Table     |──→   |Inner Page  |──→ Frame
    +------------+      |  Table     |
                        +------------+

   Logical Address: | Outer Page | Inner Page | Offset |

   32-bit example (4KB pages):
   | 10 bits | 10 bits | 12 bits |

   Only load needed inner page tables.


2. HASHED PAGE TABLE:
   - Hash page number
   - Chain for collisions
   - Good for sparse address spaces


3. INVERTED PAGE TABLE:
   - One entry per frame (not per page)
   - Entry: (process-id, page-number)
   - Smaller table, but slower lookup

   | Frame | PID | Page |
   |   0   |  P1 |   5  |
   |   1   |  P2 |   3  |
   |   2   |  P1 |   2  |
   ...


================================================================================
                    9. SEGMENTATION
================================================================================

Logical memory divided into variable-size segments.
Based on logical divisions (code, data, stack).


LOGICAL ADDRESS:
<segment-number, offset>


SEGMENT TABLE:
| Segment | Base  | Limit |
|    0    | 1400  |  400  |  ← Code segment
|    1    | 6300  |  100  |  ← Data segment
|    2    | 4300  |  1000 |  ← Stack segment


ADDRESS TRANSLATION:

1. Extract segment number and offset
2. Check offset < limit (else trap)
3. Physical address = base + offset


ADVANTAGES:
- Matches logical structure
- Different protection per segment
- Easy sharing of segments


DISADVANTAGES:
- External fragmentation
- Variable-size segments harder to manage


SEGMENTATION WITH PAGING:
Combine both - segment into pages.
Intel x86 uses this approach.

================================================================================
                    11. INTERVIEW QUESTIONS
================================================================================

Q1: What is logical vs physical address?
A: Logical: Generated by CPU, used by process.
   Physical: Actual RAM address. MMU translates between them.

Q2: What is paging?
A: Memory divided into fixed-size frames (physical) and pages (logical).
   Page table maps pages to frames. No external fragmentation.

Q3: What is internal vs external fragmentation?
A: Internal: Unused space within allocated block.
   External: Free memory scattered, can't satisfy large request.

Q4: What is TLB?
A: Translation Look-aside Buffer. Cache for page table entries.
   Speeds up address translation by avoiding memory access.

Q5: What is segmentation?
A: Memory divided into logical variable-size segments.
   Each segment has base and limit. External fragmentation possible.

Q6: Explain address translation in paging.
A: Split address into page number and offset.
   Look up page table to get frame number.
   Physical = frame × page_size + offset.

Q7: What is a multi-level page table?
A: Hierarchical page table to reduce memory usage.
   Only load needed portions. Two-level, three-level, etc.
